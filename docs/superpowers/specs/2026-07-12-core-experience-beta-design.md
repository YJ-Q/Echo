# Margin 自用内测版核心体验验收设计

日期：2026-07-12

## 目标

把现有首页、成长轨迹和留痕能力串成一条可以每天稳定使用的核心成长闭环：用户在真实倾诉中看见问题，经确认后形成成长线，完成一个足够小的实验，并能在后续记录中回看变化。

本阶段以“稳定自用”为目标，不新增功能页，不扩展心理学模型，不开发装饰性心绪功能，也不改变已经确认的三页视觉结构。

## 验收故事

固定使用“在会议中不敢完整表达”作为基线故事：

1. 用户在首页倾诉自己在会议中经常沉默，担心表达不完整。
2. Margin 通过回应帮助用户看见具体困难，但不会直接诊断或替用户下结论。
3. 当对话中出现可持续练习的方向时，首页提出一条成长线建议。
4. 只有用户明确确认后，系统才创建成长线和第一个小实验。
5. 用户在成长轨迹页查看当前理解、担心、尝试方向、节点与真实情境记录。
6. 用户完成小实验并留下结果，相关内容进入留痕。
7. 留痕页能查看最近留下、长期留下、慢慢形成和最近印记，并能按既定的固定槽位交换聚焦内容。
8. 应用重启后，对话、成长线、实验状态、情境记录和留痕仍然存在。

## 产品边界

- Margin 提供陪伴、复述、澄清和行动建议，不进行医学诊断，也不把推测描述成事实。
- AI 观察只有在用户确认后才能成为成长线、长期理解或主动保存的留痕。
- 一次只推动一个足够小的实验，避免同时生成多个任务造成压力。
- 首页负责倾诉与发现，成长轨迹负责练习与记录，留痕负责回看与校准；不增加第四张功能页。
- 成长印记只表达真实发生过的变化，不作为积分、等级或连续签到系统。

## 验收架构

采用“确定性自动化基线 + 真实供应商人工验收”的混合方式。

### 确定性基线

使用固定输入、模拟供应商响应和隔离数据库验证状态转换。测试必须能够重复运行，并覆盖创建、重复提交、完成、持久化和页面视图模型。供应商网络状态不能影响这组测试的结果。

### 真实使用验收

使用桌面应用和真实供应商配置完成一次完整故事。真实验收关注回应质量、等待反馈、错误提示、滚动裁切和重启恢复，不使用自动回退结果冒充供应商联调成功。

### 阶段边界

核心闭环验收完成后，供应商实机矩阵和 Windows 安装包分别进入独立阶段。第一阶段可以修复阻断核心故事的供应商或桌面问题，但不在本规格中扩展新的供应商和打包能力。

## 功能节点与完成标准

### 1. 首页对话与发现

- 用户消息保持右对齐，Margin 消息保持左对齐。
- 发送期间有明确但克制的等待状态；重复提交不会创建两轮相同对话。
- 右侧“我看见的”“我注意到”“还可以继续看看”来自当前可解释的数据状态。
- 成长线建议必须包含主题、为什么值得继续和一个可拒绝的确认入口。
- 用户拒绝建议后继续普通对话，不创建成长线。

### 2. 成长线确认与创建

- 同一轮确认最多创建一条成长线。
- 创建结果包含明确主题、当前节点和一个可在短时间内完成的小实验。
- 重复点击确认返回同一结果或被安全忽略，不产生重复会话、节点或行动。
- 创建失败时保留当前对话，并提供可重试提示。

### 3. 成长轨迹推进

- “我现在的理解、我注意到、可能担心、想试试看”与当前成长线一致。
- 路径只展示上一个已完成节点、当前节点和下一个未开始节点；未开始节点不可进入记录。
- 本周小实验显示状态、操作入口和结果记录入口。
- 真实情境记录一次显示一天，周与日期切换不会丢失当前选择；长内容只在记录区域滚动。
- 完成实验后，当前节点、记录和首页摘要在刷新后保持一致。

### 4. 留痕回看与校准

- 默认左侧为最近留下，右侧为慢慢形成与最近印记。
- 长期留下只替换留痕记录槽位；慢慢形成和印记通过固定槽位交换成为左侧重点。
- 用户可以修正不准确的长期理解，修正后立即显示用户版本。
- 保存、取消保存和整理操作具备明确结果，不会因重复点击产生冲突状态。
- 印记只能由已发生的后端事件解锁，刷新或重复提交不会重复解锁。

### 5. 重启与恢复

- 关闭桌面应用后重新打开，数据库仍位于既定用户数据路径。
- 对话、成长线、步骤状态、情境记录、长期理解、留痕和印记保持可读。
- 页面可以从恢复后的数据重新构建，不依赖上一次渲染器内存状态。

## 异常与边界状态

- 空数据库：三张功能页都有与产品语气一致的空状态和下一步入口。
- 无效密钥：显示供应商与稳定错误类别，不显示密钥、请求头或上游响应正文。
- 超时或断网：明确说明本次未完成；允许重试；本地回退不得被标记为供应商成功。
- 不完整 AI 输出：不创建结构不完整的成长线或行动，保留普通对话结果并提示重新整理。
- 重复操作：创建、确认、完成、保存和印记确认均保持幂等。
- 长文本：消息、理解、记录和留痕在各自内容区滚动，不扩大或覆盖笔记本外壳。
- 小窗口：关键标题、当前操作和返回入口不得被裁切；不要求为手机尺寸重新设计。

## 数据流与一致性

后端数据库是长期状态的唯一事实来源。前端可以维护临时选择、展开状态和加载状态，但不能把未落库的渲染器状态当作已完成结果。

核心状态顺序为：

```text
conversation
  -> confirmed learning suggestion
  -> learning session + current step
  -> experiment result + learning event
  -> memory / profile synthesis
  -> achievement evaluation
  -> page-ready state reload
```

每次写操作成功后，应重新读取相关页面数据或使用同一后端响应更新视图。跨页面看到的主题、步骤状态和记录文本必须来自同一持久化结果。

## 测试策略

### 自动化

- 服务层测试状态转换、幂等与错误分类。
- 路由测试验证 API 合约、无效输入和页面就绪数据结构。
- 视图模型测试验证首页、成长轨迹和留痕对同一数据的解释一致。
- 结构与视觉系统测试防止整页背景覆盖、右侧旧分页和印记跨页回归。
- 持久化测试使用临时数据库，完成写入、关闭、重新打开和读取。

### 桌面人工验收

- 使用 4:3 桌面窗口走完固定故事。
- 分别检查默认数据、空数据、长文本和供应商失败状态。
- 重启应用后重复检查三张功能页。
- 记录阻断问题、可接受问题和延期问题；只有阻断问题清零才进入供应商实机矩阵。

## 发布门槛

第一阶段完成必须同时满足：

- 固定验收故事可以从首页连续走到留痕，无死路和人工改数据库步骤。
- 用户确认之前不会创建成长线或长期理解。
- 重复操作不会产生重复数据。
- 重启后核心状态完整恢复。
- 三张功能页没有新增整页覆盖层、越界纸张或关键控件裁切。
- 自动化测试、TypeScript 检查和前端生产构建全部通过。
- 已知问题列表中不存在阻断核心故事的问题。
